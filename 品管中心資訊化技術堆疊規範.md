# 品管中心資訊化專案 — 制式化技術堆疊規範

> **版本：** 1.0  
> **建立日期：** 2026-03-22  
> **適用範圍：** 新竹臺大分院品質管理中心所有自建資訊化專案  
> **文件性質：** 每個新專案啟動前，必須依據本規範進行技術選型與架構設計

---

## 一、規範目的

品管中心的資訊化專案過去因為各案獨立發展，導致技術堆疊分散（R Shiny、Excel、Python 各自為政），造成三個核心問題：

1. **維護人力分散** — 每套系統需要不同技能的維護者，一旦人員異動就可能斷裂
2. **資料無法互通** — 各系統的資料格式和儲存方式不同，跨系統查詢極為困難
3. **新人接手困難** — 缺乏一致的開發慣例，交接成本極高

本規範的目標是：**讓所有專案共用同一套技術語言、同一套資料庫、同一套部署流程**，使任何一位開發者都能快速理解並接手任何一個專案。

## 三、各層級詳細規範

### 3.1 程式語言：Python 3.12+

**為什麼選 Python？**

- 建言資料庫已採用 Python + PostgreSQL + Redis，證實此組合在品管中心場景下穩定可行
- Python 在 Web 開發（FastAPI / Django）、資料分析（pandas / numpy）、AI/NLP（用於未來的事件自動分類、文本分析）三個面向的生態系最完整
- 醫院內部若有統計分析需求，Python 的 scipy、statsmodels 可覆蓋絕大多數場景

**關於 R 的定位：**

- 現有的「病人安全文化調查 R Shiny 儀表板」繼續維護運行，不需砍掉重練
- R 保留用於特殊統計分析（如問卷信效度分析 — Cronbach's α、CFA 等），但僅作為資料處理腳本，不再開設新的 R Shiny Web 應用
- 未來新的儀表板需求一律使用 Python 生態系

**編碼規範：**

```
- 程式碼風格遵循 PEP 8
- 使用 type hints（Python 3.12 的泛型語法）
- 使用 ruff 作為 linter 和 formatter
- 每個專案根目錄必須有 pyproject.toml 管理依賴
- 依賴管理使用 pip + requirements.txt 或 poetry
```

### 3.2 Web 框架選型指引

啟動新專案時，根據以下決策樹選擇框架：

```
新專案需求評估
│
├─ 是否需要完整的後台管理介面（Admin）？
│  ├─ 是 → 是否有複雜的使用者角色權限？
│  │       ├─ 是 → Django + Django REST Framework
│  │       └─ 否 → Django（使用內建 Admin）
│  │
│  └─ 否 → 是否主要提供 API 給前端或其他系統呼叫？
│           ├─ 是 → FastAPI
│           └─ 否 → 是否為內部分析儀表板？
│                    ├─ 是 → Streamlit（單獨部署）
│                    └─ 否 → FastAPI + React 前端
```

**各專案的建議框架對照：**

| 現有/規劃中專案 | 建議框架 | 理由 |
|---------------|---------|------|
| 建言資料庫 | FastAPI（維持現狀） | 已運行中，API 導向 |
| QIP 監測指標 | Django | 需要後台管理介面供品管人員操作 |
| 醫學中心五大任務 | Django | 需要權限控管、進度追蹤後台 |
| 醫院資料日報表 | FastAPI + Streamlit | API 取數 + 視覺化儀表板 |
| 病人安全文化調查 | R Shiny（維持現狀） | 已完成，未來新需求改用 Python |
| 異常事件通報系統（未來） | Django | 複雜表單、工作流程、權限控管 |
| QCC 改善專案追蹤（未來） | Django | PDCA 進度管理、後台操作 |
| 滿意度調查系統（未來） | FastAPI + React | 面向全院的填答前端 |

### 3.3 資料庫：PostgreSQL 16+

**核心原則：所有專案共用同一個 PostgreSQL 實例，以 schema 隔離。**

```sql
-- 每個專案使用獨立的 schema
CREATE SCHEMA qip;          -- QIP 監測指標
CREATE SCHEMA suggestion;   -- 建言資料庫
CREATE SCHEMA five_tasks;   -- 醫學中心五大任務
CREATE SCHEMA daily_report; -- 醫院資料日報表
CREATE SCHEMA incident;     -- 異常事件通報（未來）
CREATE SCHEMA survey;       -- 滿意度/文化調查（未來）
CREATE SCHEMA qcc;          -- QCC 改善專案追蹤（未來）

-- 共用的參照資料放在 public schema
-- 例如：科室代碼表、人員基本資料表
CREATE TABLE public.departments (...);
CREATE TABLE public.staff (...);
```

**為什麼用 PostgreSQL 而非 MySQL？**

- JSONB 欄位原生支援，適合儲存半結構化的品質資料（通報表單、問卷回答結構各異）
- 全文檢索內建（`tsvector`），對建言、通報事件的文字搜尋非常有用
- 與 Django ORM 和 SQLAlchemy 的相容性最佳
- Row-Level Security（RLS）可實現科室層級的資料權限控管

**資料庫命名慣例：**

```
- 表名：小寫英文 + 底線分隔，複數形式（如 incidents, surveys）
- 欄位名：小寫英文 + 底線分隔（如 created_at, department_id）
- 主鍵：統一使用 id（自增 BIGINT 或 UUID，依場景選擇）
- 時間欄位：所有表必須包含 created_at 和 updated_at
- 軟刪除：需要軟刪除的表加上 deleted_at 欄位
```

### 3.4 快取與任務佇列：Redis 7.0+

**用途分類：**

| 用途 | Redis DB | 說明 |
|------|---------|------|
| Session 快取 | DB 0 | Web 應用的使用者 session |
| 應用快取 | DB 1 | API 回應快取、指標計算結果快取 |
| Celery Broker | DB 2 | 非同步任務佇列的訊息代理 |
| Celery Result | DB 3 | 非同步任務執行結果儲存 |

**典型的非同步任務場景：**

- QIP 指標從 HIS 定期撈取並計算（排程任務）
- 日報表每日自動生成並寄送（排程任務）
- 異常事件通報後自動發送通知（事件驅動任務）
- 大量問卷資料匯入與統計（耗時任務）

### 3.5 前端技術

**Streamlit — 內部分析用途：**

```
適用場景：
- 品管人員自己看的分析儀表板
- 原型驗證（先用 Streamlit 快速做出來，確認需求後再決定是否升級為 React）
- 資料探索工具

不適用場景：
- 需要全院數百人同時使用的系統
- 需要複雜互動（拖拉、多步驟表單）的介面
```

**React + TypeScript — 正式系統：**

```
適用場景：
- 異常事件通報系統（全院使用）
- 滿意度調查填答介面
- 任何需要在手機瀏覽器上良好運作的系統

UI 框架：Ant Design 或 Material UI（選定一套後全部專案統一）
狀態管理：Zustand 或 React Query（依複雜度選擇）
建置工具：Vite
```

### 3.6 容器化部署：Docker + docker-compose

**所有專案必須提供以下檔案：**

```
project-root/
├── Dockerfile              # 應用程式容器定義
├── docker-compose.yml      # 本機開發環境編排
├── docker-compose.prod.yml # 正式環境編排（覆蓋開發設定）
├── .env.example            # 環境變數範本（不含機敏資料）
├── .dockerignore           # 排除不必要檔案
└── ...
```

**docker-compose 共用服務模板：**

```yaml
# 所有專案共用的基礎設施服務
services:
  postgres:
    image: postgres:16-alpine
    volumes:
      - postgres_data:/var/lib/postgresql/data
    environment:
      POSTGRES_DB: qm_center
      POSTGRES_USER: ${DB_USER}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    ports:
      - "5432:5432"

  redis:
    image: redis:7-alpine
    volumes:
      - redis_data:/data
    ports:
      - "6379:6379"
```

**映像檔命名慣例：**

```
qm-center/<專案名稱>:<版本號>

範例：
qm-center/suggestion-api:1.2.0    # 建言資料庫
qm-center/qip-dashboard:0.5.0     # QIP 監測指標
qm-center/daily-report:1.0.0      # 日報表
```

## 五、共用服務與中介層

### 5.1 統一認證機制

所有 Web 應用採用統一的認證方式，避免每個系統各自管理帳號密碼：

```
優先順序：
1. 如果醫院有 LDAP/AD → 串接院內 Active Directory
2. 如果無法串接 → 建立共用的認證服務（OAuth2 / JWT）
3. 所有專案透過共用認證服務驗證使用者身份
```

### 5.2 共用資料表

以下資料在 PostgreSQL 的 `public` schema 中維護，所有專案共用：

```
public.departments    — 科室代碼與名稱
public.staff          — 員工基本資料（工號、姓名、科室、職稱）
public.audit_log      — 統一的操作稽核紀錄
public.notifications  — 統一的通知佇列
```

### 5.3 API 設計慣例

```
- RESTful 風格
- 路徑命名：小寫 + 連字號（如 /api/v1/quality-indicators）
- 版本號放在路徑中（/api/v1/...）
- 回應格式統一為 JSON
- 分頁格式統一：{ "data": [...], "total": 100, "page": 1, "page_size": 20 }
- 錯誤回應格式統一：{ "error": { "code": "NOT_FOUND", "message": "..." } }
- 日期時間格式統一：ISO 8601（如 2026-03-22T08:30:00+08:00）
```

## 六、開發流程規範

### 6.1 版本控制

```
- 每個專案獨立一個 Git repository
- 分支策略：main（正式）→ develop（開發）→ feature/* （功能分支）
- Commit message 格式：<type>: <description>
  - feat: 新功能
  - fix: 修正錯誤
  - docs: 文件更新
  - refactor: 重構
  - test: 測試
```

### 6.2 環境管理

```
每個專案維護三個環境：
1. local   — 開發者本機（docker-compose up）
2. staging — 測試環境（部署前驗證）
3. prod    — 正式環境

環境變數管理：
- 機敏資料（密碼、金鑰）絕不進入 Git
- 使用 .env 檔案管理，.env.example 提供範本
- 正式環境使用環境變數注入或 secrets manager
```

### 6.3 資料庫遷移

```
- FastAPI 專案使用 Alembic
- Django 專案使用內建 migrations
- 每次 schema 變更必須產生遷移檔案並提交至 Git
- 禁止直接在正式環境手動修改資料庫結構
```

### 6.4 測試要求

```
- 每個專案至少覆蓋核心業務邏輯的單元測試
- API 端點必須有整合測試
- 測試框架統一使用 pytest
- 測試資料庫使用 PostgreSQL（不要用 SQLite 替代，避免方言差異）
```

## 七、現有專案遷移優先順序

根據急迫性和影響範圍，建議按以下順序將現有專案遷移到制式化堆疊：

| 優先序 | 專案 | 遷移內容 | 預估工作量 |
|-------|------|---------|-----------|
| 1 | QIP 監測指標 | Excel → Django + PostgreSQL + Streamlit 儀表板 | 中（2-4 週） |
| 2 | 醫院資料日報表 | 現有流程 → FastAPI + PostgreSQL + Streamlit | 中（2-3 週） |
| 3 | 醫學中心五大任務 | 現有流程 → Django + PostgreSQL | 中（2-4 週） |
| 4 | 建言資料庫 | 已符合規範，微調即可 | 小（1 週內） |
| 5 | 病人安全文化調查 | R Shiny 維持運行，資料層接入 PostgreSQL | 小（1-2 週） |

## 八、未來擴充預留

### 8.1 AI / NLP 整合

當品管中心未來導入 AI 功能時（如異常事件自動分類、建言情感分析、RCA 輔助），Python 生態系可直接使用：

```
- 文字分析：jieba（中文斷詞）、transformers（預訓練模型）
- 機器學習：scikit-learn、XGBoost
- LLM 整合：Anthropic API / OpenAI API
- 向量資料庫：pgvector（PostgreSQL 擴充套件，免額外部署）
```

### 8.2 BI 工具整合

如果未來需要讓管理階層使用 Power BI 或 Tableau：

```
- PostgreSQL 可直接作為 Power BI / Tableau 的資料源
- 在 PostgreSQL 中建立 views 作為 BI 工具的取數介面
- 不需要額外的資料倉儲，PostgreSQL 即可勝任
```

### 8.3 行動裝置支援

```
- React 前端採用 RWD 設計，確保手機瀏覽器可正常操作
- 若未來需要原生 App，可使用 React Native 共用前端技能
- 推播通知可透過 LINE Notify 或 Web Push API
```
