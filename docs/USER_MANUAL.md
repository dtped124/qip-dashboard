## 1. 系統概述

### 1.1 系統目的

本系統為新竹生醫園區醫院體系（含新竹、竹北、竹東三院區）品管中心開發，提供品質指標的**持續性監測**，取代以往每三年一次評鑑前才緊急彙整 Excel 的模式。

### 1.2 核心理念

從「每三年一次評鑑大考」→「每月小測驗」：每月匯入當月資料、即時看到異常、主動改善。

### 1.3 關鍵角色與場景

| 使用者 | 場景 | 典型時長 | 核心問題 |
|--------|------|---------|---------|
| 品管人員 | 每月匯入新數據、檢視異常 | 5-10 分鐘 | 這個月哪些指標出問題？ |
| 委員會委員 | 每季審視改善進度 | 30 秒摘要 + 5 分鐘深入 | 這一季整體進步還是退步？ |
| 院級主管 | 策略性資源配置 | 2 秒結論 | 要把資源投在哪裡？ |

### 1.4 四層決策路徑

1. **2 秒**：看燈號分佈、總異常數
2. **30 秒**：哪幾支紅燈
3. **2 分鐘**：指標詳情（管制圖、趨勢、同儕）
4. **5 分鐘**：AI 深度分析、改善建議

## 2. 系統安裝與啟動

### 2.1 系統需求
- Node.js 18+、Python 3.11+
- 建議瀏覽器：**Chrome**、Edge

```bash

npm install
npm run dev            # http://localhost:3000

# 後端（Docker）

docker compose up -d
```

- 登入識別：**員工編號**（`employee_id`）
- 初始密碼：員工編號，首次登入強制更改

## 3. 登入與權限

### 3.1 三種角色

| 角色 | 說明 |
|------|------|
| **admin**（系統管理員） | 完整權限，管理帳號、指派、截止日、匯入、匯出 |
| **reviewer**（品管審核） | 審核月報、核准定稿、個案排除審核 |

1. 訪問 `http://your-domain/entry/login`
2. 輸入**員工編號** + 密碼
3. 首次登入須更改密碼
4. 登入後依角色跳轉：reporter → 我的任務、reviewer → 審核總覽、admin → 管理後台

## 4. 儀表板首頁

### 4.1 總覽

首頁（`/`）顯示：
- **院區切換**：新竹 / 竹北 / 竹東
- **類別分組**：10 大類別（HA01-HA10）
- **三種視圖切換**：
  - 卡片視圖（預設）：每指標一張卡片，顯示最新值、Sparkline、狀態燈號
  - 狀態矩陣：指標 × 月份的燈號網格
  - 表格視圖：傳統表格列

### 4.2 指標卡片資訊

| 區塊 | 內容 |
|------|------|
| 指標名稱 + 代碼 | 如 `HA01-01 住院死亡率` |
| 燈號標記 | 卓越 / 良好 / 留意 / 注意 / 警示 / 監測中 |
| 最新值 + 期間 | 如 `2.1% (115.03)` |
| Sparkline | 最近 24 個月迷你趨勢圖 |
| 年均值 | 分母加權平均 |
| 同儕值 | TCPI 或區域/地區標竿 |
| 趨勢箭頭 | 上升 / 下降 / 持平 |
| 異常標籤 | 「管制圖 2σ 警示」「高於同儕 15%」等 |

### 4.3 六級燈號系統

| 燈號 | 顏色 | 含意 |
|------|------|------|
| 卓越 | 藍 | 優於同儕且管制圖穩定 |
| 良好 | 綠 | 管制圖穩定、同儕內 |
| 留意 | 黃 | 2σ 觀察、或輕微高於同儕 |
| 注意 | 橘 | 2σ 異常 / 月增減顯著 |
| 警示 | 紅 | 超出 UCL/LCL、或連續惡化 |
| 監測中 | 灰 | 資料不足或無判定 |

### 4.4 篩選功能

- **類別**：僅看某個面向
- **搜尋**：指標代碼 / 名稱
- **僅看異常**：隱藏良好與卓越

## 6. SPC 管制圖與異常判讀

### 6.1 管制圖類型

| 類型 | 適用資料 | 範例指標 |
|------|---------|---------|
| **P 圖** | 二項比率（百分比） | 住院死亡率、再住院率 |
| **U 圖** | Poisson 密度（‰） | VAP、CAUTI、CLABSI |
| **I-MR 圖** | 單點連續資料 | 平均住院日、件數 |

系統**自動依資料性質選型**，無需手動指定。

### 6.2 管制界線

- **CL**（中心線）：歷史平均（p̄ / ū / X̄）
- **UCL / LCL**（3σ 管制界線）
- **UCL2 / LCL2**（2σ 警示界線）
- **Variable limits**：當分母 n 每月變化大時，每點有自己的 UCL/LCL

### 6.3 Nelson Rule 異常偵測

| 規則 | 說明 | 嚴重度 |
|------|------|-------|
| Rule 1 | 單點超出 3σ | 警示（紅） |
| Rule 2 | 連續 9 點同側 | 注意（橘） |
| Rule 3 | 連續 6 點遞增或遞減 | 注意 |
| Rule 4 | 連續 2/3 點在 2σ 外 | 留意（黃） |
| ... | 共支援數條 Nelson Rule | |

### 6.4 其他異常機制

- **月增減**：對比前月變動 > 閾值（依方向性判定 favorable / unfavorable）
- **同儕比較**：
  - 越低越好：> 同儕值 × 1.10 → unfavorable
  - 越高越好：< 同儕值 × 0.90 → unfavorable
  - 監測：|差異| > 20% → 留意

### 6.5 跳過 SPC 的指標

部分純計數型指標（如「給藥錯誤件數」）不適合畫管制圖，系統透過 `SKIP_SPC_INDICATORS` 清單跳過，僅用月增減 + 同儕偵測。

## 7. 跨院區季度分析

路徑：`/cross-campus`

### 7.1 用途

委員會會議用：一次比較三院區在所有指標的季度表現，不必逐院區切換。

### 7.2 顯示內容

- 每指標一列，三個院區的季度資料並列
- 季度管制圖（重新計算 CL/UCL/LCL）
- 標註改善 / 惡化趨勢
- 「先報喜再報憂」排序：先展示進步指標

### 7.3 季度彙總邏輯

| 資料性質 | 彙總方式 |
|---------|---------|
| `binomial_rate` | 分子加總 / 分母加總 → 重算比率 |
| `poisson_rate` | 同上 |
| `continuous`（加總型） | 月值加總 |
| `continuous`（平均型） | 加權平均 |

## 8. Excel 資料匯入

路徑：`/import`（admin / reviewer）

### 8.1 支援格式

- 檔案：`.xls`、`.xlsx`
- 工作表命名：`111年竹北`、`112年竹東`、`115年新竹` 等
- 典型結構：17 張工作表（5 年 × 3 院區 + 少數特殊）

### 8.2 匯入流程

1. **上傳檔案** → 系統自動解析
2. **預覽摘要**：
   - 新增：X 筆
   - 更新：X 筆
   - 未變更：X 筆
   - 錯誤 / 警告：列表
3. **警告展開**：點擊可看單筆問題（分母為 0、指標名稱無法比對等）
4. **直接修正**：每條警告旁有「修正並移除」按鈕，輸入正確值即可
5. **名稱比對**：若 Excel 名稱與指標對不上，系統提示使用者確認；確認後寫入 `matching_rules`，下次自動比對

### 8.3 新竹院區特殊處理

新竹（醫學中心）的 Excel 格式與竹北/竹東略有不同，由 `hsinchu_parser.py` 另行解析。

### 8.4 匯入紀錄

路徑：`/settings`（匯入紀錄區塊）
- 檔名、大小、上傳時間
- 各院區各月異動統計
- 錯誤清單

## 9. TCPI 標竿管理

路徑：`/settings/tcpi`（admin）

### 9.1 TCPI 標竿結構

每筆 TCPI 含三種醫院層級的同儕值：
- 醫學中心（`medical_center`）→ 新竹院區用
- 區域醫院（`regional_hospital`）→ 竹北院區用
- 地區醫院（`district_hospital`）→ 竹東院區用

### 9.2 匯入 TCPI

1. 下載 TCPI Excel 範本
2. 填入當年度同儕值
3. 上傳 → 預覽 → 確認
4. ⚠️ **注意**：匯入為**覆蓋**模式，會清除所有舊 TCPI 資料

### 9.3 標竿選用優先序

```
1. YearlySummary.benchmark_regional / benchmark_district（若該指標有年度彙總）
   ↓ 若無
2. TCPIBenchmark（依院區對應到三種層級）
```

## 10. 挑戰平均值模式

### 10.1 用途

預設管制圖 CL = 歷史平均（p̄ / ū / X̄）。當品管中心想訂下**改善目標**（例如將住院死亡率從 2.1% 降至 1.5%）時，可啟用**挑戰平均值模式**，以 1.5% 取代 CL，UCL/LCL 隨之重算。

此功能對應吳文祥教授 SPC 範本同名機制。

### 10.2 啟用方式

路徑：`/settings/indicators`（admin）

1. 找到目標指標
2. 開啟「挑戰平均值模式」開關
3. 輸入「挑戰目標值」（如 `1.5`）
4. 系統立即：
   - 更新 `Indicator.target_mode = True`
   - 更新 `Indicator.target_value`
   - **自動重算**該指標所有院區的 `Alert`

### 10.3 視覺呈現

管制圖上方會標註「挑戰平均值模式：目標 1.5%」，CL 線改用紅色虛線區別。

## 11. 填報系統

### 11.1 我的任務

reporter 登入後進入 `/entry`：

- **待辦列表**：所有被指派且狀態為 `unfilled` / `draft` 的月報
- 每筆顯示：院區、年月、面向、截止日、進度（已填 N / 共 M 項）
- **逾期標示**：超過當月截止日仍未送審 → 紅色警告

### 11.2 填寫表單

點擊某面向 → `/entry/[category]`

#### 表單區塊
- 標題列：院區、年月、面向、截止日
- 每個指標一個區塊：
  - 指標名稱 + 計算公式提示
  - 分子 / 分母欄位
  - 自動計算值（唯讀）
  - 備註欄
  - （HA10）子類別明細 13 項
  - （個案清單路徑）連結至個案清單頁

#### 操作按鈕
- **儲存草稿**：隨時可存，不變更狀態
- **送審**：狀態 `draft → submitted`，送至品管中心

### 11.3 填報截止日

- 預設每月 10 號
- 管理員可在 `/entry/admin` 逐月調整（春節、颱風等延長）

### 11.4 修改紀錄

每次送審後若需修改，審核者直接在 `/api/review/edit-entry` 修改會自動寫入 `EntryAuditLog`，保留：
- 欄位名
- 原值 → 新值
- 修改者、時間、理由

## 12. 審核系統

reviewer 登入後進入 `/entry/review`

### 12.1 審核總覽

依年月 + 面向 + 院區 + 狀態篩選的矩陣：

| 院區 | HA01 | HA02 | HA03 | ... |
|------|------|------|------|-----|
| 新竹 | ✅核准 | ⏳待審 | ❌退回 | ... |
| 竹北 | 🟢定稿 | ✅核准 | ⏳待審 | ... |
| 竹東 | ⏳待審 | ⏳待審 | 🔒未填 | ... |

### 12.2 狀態流程

```
未填（unfilled）
  → 草稿（draft）
    → 已送審（submitted）
      ↙                   ↘
  退回（draft）     已核准（approved）
                          ↓
                   已送出（finalized，鎖定）
                          ↓
                    （admin 可 unlock）
```

### 12.3 審核操作

- **查看詳情**：完整 IndicatorEntry + 歷次修改紀錄
- **核准**：`submitted → approved`
- **退回**：`submitted → draft`，需填寫退回理由
- **直接修正**：可在審核過程直接改值（自動寫入 AuditLog）
- **定稿**：`approved → finalized`，資料會同步寫入 `data_points` 表供儀表板使用
- **解鎖**：`finalized → approved`（限 admin，需填寫理由）

## 13. 個案清單審查

對於 `entry_mode = "case_list"` 的指標（如某些特定疾病追蹤），填報流程需：

### 13.1 個案清單

路徑：`/entry/case-list?entry_id=xxx`

- 顯示從 HIS 抓取的**分子群**與**分母群**病例
- 每筆含病歷號、住院/出院日、結果、科別等 JSON 原始資料
- 篩選：已排除 / 未排除、分子 / 分母

### 13.2 申請排除

reporter 可勾選要排除的個案：
1. 選擇排除理由（從預設清單 `ExclusionReason`）
2. 填寫補充說明（選填）
3. 送出 → 狀態標記 `is_excluded=True`，但需等 reviewer 審核

### 13.3 品管審核排除

reviewer 在 `/entry/review` 的排除審核分頁：
- 看所有待審核的排除申請
- 核准 / 駁回，寫入 `reviewer_approved`、`reviewer_note`
- 僅**核准**的排除會從最終分子 / 分母扣除

### 13.4 統計呈現

```
原始分子：25
原始分母：800
排除分子：3（已核准）
排除分母：12（已核准）
最終分子：22
最終分母：788
指標值：2.79%
```

## 15. 常見問題

### Q1: 為什麼同一指標不同院區的標竿值不同？

**A**: 系統依院區類型自動選用：
- 新竹（醫學中心）→ TCPI `medical_center`
- 竹北（區域醫院）→ TCPI `regional_hospital`
- 竹東（地區醫院）→ TCPI `district_hospital`

### Q2: 年均值為什麼跟 Excel 算出來不一樣？

**A**: 系統採用**分母加權平均**，不是簡單平均。例如某指標分母每月差異大時，加權平均更能反映實際比率。公式：

```
年均值 = Σ(月值 × 月分母) / Σ(月分母)
```

### Q3: 管制圖沒顯示怎麼辦？

**A**: 以下情況會跳過管制圖：
1. 該指標在 `SKIP_SPC_INDICATORS` 清單中（純計數型）
2. 有效資料點 < 12 個（建議至少一年）
3. 分母全為 0（P/U 圖無法計算）

可改看月增減與同儕比較的異常列表。

### Q4: 挑戰平均值開了但異常沒重算？

**A**: 系統已自動呼叫 `_refresh_indicator_alerts()`。若仍看到舊警示：
1. 確認存檔成功（API 回應 200）
2. 重新整理頁面
3. 檢查 `target_value` 是否為有效數字

### Q5: Excel 匯入後資料點沒變多，但顯示「更新：XX 筆」是什麼意思？

**A**: `(indicator, campus, year, month)` unique 已存在時視為更新，不是新增。系統記錄：
- 新增：原本沒有的資料點
- 更新：數值不同的既有資料點
- 未變更：數值完全相同的資料點（省去 DB 寫入）

### Q6: 個案清單的排除為什麼沒有立即生效？

**A**: 排除流程設計為**二階段**：
1. reporter 提出排除 → `is_excluded=True`，但 `reviewer_approved=None`
2. reviewer 核准 → `reviewer_approved=True`，才從最終分子/分母扣除

未核准的排除不會影響指標值。

### Q7: 送審後發現錯誤怎麼辦？

**A**: 三條路：
1. **退回重填**：reviewer 按「退回」，狀態回 `draft`，reporter 改完再送審
2. **審核者直接改**：reviewer 有權限時可在審核介面直接修改（會寫入 AuditLog）
3. **已定稿了？**：請 admin 解鎖（`finalized → approved`）再修

### Q8: 為什麼我看不到某個院區的資料？

**A**: 部分指標只適用特定院區（見 `Indicator.campuses`）：
- 某些加護指標在竹東（地區醫院）可能不存在（無 ICU 或規模不足）
- `HSINCHU_ONLY` 指標只在新竹院區才顯示

### Q9: 填報系統的截止日逾期會怎樣？

**A**: 目前僅顯示紅色逾期標籤，不阻擋送審。後續規劃：
- 逾期達 3 日自動 Email 提醒
- 月底未送審則 `is_late=True` 紀錄備查

### Q10: Portable 模式跟伺服器模式的差異？

**A**:
| 模式 | 資料來源 | 可否編輯 | 適用 |
|------|---------|---------|------|
| 伺服器（Docker） | PostgreSQL 即時 | ✅ 可匯入、填報、審核 | 品管中心日常使用 |
| Portable（Static） | 匯出的 JSON + IndexedDB | ❌ 僅讀取 | 交付委員、離線瀏覽 |

Portable 版本以 `STATIC_EXPORT=true npm run build` 產生 `out/` 資料夾，可用任何靜態伺服器開啟。
