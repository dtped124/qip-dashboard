## 二、權限控管系統

### 2.1 角色定義

| 角色名稱 | 英文識別 | 對象 | 權限陣列 |
|----------|----------|------|----------|
| 系統管理員 | `admin` | 品管中心承辦人 | `["all"]` |
| 督導長官 | `supervisor` | 科室主管 / 督導人員 | `["view", "assess", "approve", "view_restricted"]` ← 可看全部一般與限制條文，但無 admin 操作權限 |
| 填表人 | `filler` | 各條文負責填寫人員 | `["view", "edit", "upload"]`　view = 可看所有一般條文 |
| 唯讀 | `readonly` | 只能查看、不能修改 | `["view"]` |

**權限值說明：**

| 權限值 | 說明 |
|--------|------|
| `view` | 基本查看（所有一般條文） |
| `edit` | 編輯自己負責的表格資料（含自己被指派的限制條文） |
| `upload` | 上傳佐證資料 |
| `assess` | 填寫自評表、審核操作 |
| `approve` | 核准操作 |
| `view_restricted` | 查看所有限制性條文（supervisor / admin） |
| `all` | 管理員全部權限（包含上述所有 + 系統管理操作） |

### 2.2 三層防線模型

```
第一層：身份驗證（Authentication）
  └─ Flask-Login：確認使用者已登入
  └─ 帳號未啟用 / 鎖定 → 拒絕

第二層：角色授權（Role Authorization）
  └─ 裝飾器：@permission_required / @admin_required / @supervisor_required
  └─ 確認 Role.permissions 包含對應權限鍵

第三層：資料範圍（Data Scope）
  └─ can_edit_table(user, table)：比對填表人名稱
  └─ can_view_table(user, table)：比對督導範圍 + 限制條文控制
  └─ get_user_accessible_tables(user, query)：過濾查詢集
```

### 2.3 裝飾器用法

| 裝飾器 | 需要角色 | 用途 |
|--------|----------|------|
| `@permission_required('edit')` | filler 以上 | 需 edit 權限的路由（資料編輯） |
| `@permission_required('assess')` | supervisor 以上 | 需 assess 權限的路由（自評審核） |
| `@admin_required` | admin 專用 | 使用者管理、條文設定等管理後台路由 |
| `@supervisor_required` | supervisor 以上 | 督導專屬功能路由 |

> API 路由（`/api/...`、`Accept: application/json`）失敗回傳 JSON `{"success": false, "error": "..."}` + HTTP 403；頁面路由失敗則 redirect 到 dashboard。

### 2.4 資料範圍函數

#### `can_edit_table(user, table) → bool`

| 條件 | 回傳結果 |
|------|----------|
| `admin` 角色 | 永遠 True |
| `supervisor` / `readonly` 角色 | 永遠 False |
| `filler` + 名稱精確匹配（部門 + 姓名） | **True**（含限制條文：filler 可編輯自己負責的限制條文） |
| `filler` + 僅姓名匹配（不含部門） | True（記錄 warning log，待 v1.8.0 外鍵改造後取代） |
| 帳號停用或鎖定中 | 永遠 False |

#### `can_view_table(user, table) → bool`

| 角色 | 查看條件 |
|------|----------|
| `admin` | 全部可查看 |
| `supervisor` | 一般條文：全部可查看；限制條文：全部可查看；不可執行人員匯入或系統設定（admin_required 路由拒絕） |
| `filler` | 一般條文：可查看全部（含他人負責的）；限制條文：只能查看並編輯自己被指派的（名稱匹配），其他人的限制條文不可見 |
| `readonly` | 非限制條文皆可查看 |
| 限制條文存取規則 | supervisor / admin → 可查看全部限制條文；filler → 僅限自己被指派的限制條文（can_edit_table 回 True 才可見）；readonly → 完全不可見 |

### 2.5 權限矩陣快速查閱

| 操作 | admin | supervisor | filler | readonly | 說明 |
|------|:-----:|:----------:|:------:|:--------:|------|
| 查看一般條文表格 | ✅ | ✅ 全部 | ✅ 全部 | ✅ | supervisor/filler 皆可看所有一般條文 |
| 查看限制條文（RCA/經費） | ✅ | ✅ | ✅ 自己負責的 | ❌ | filler 只能看自己被指派的 |
| 編輯表格資料 | ✅ | ❌ | ✅ 自己的 | ❌ | 查看全部，但只能編輯負責的 |
| 上傳佐證資料 | ✅ | ❌ | ✅ | ❌ | |
| 填寫自評表 | ✅ | ✅ | ❌ | ❌ | |
| 送出督導回饋意見 | ✅ | ✅ | ❌ | ❌ | TableFeedback |
| 核准操作 | ✅ | ✅ | ❌ | ❌ | |
| 管理使用者 / Plan C | ✅ | ❌ | ❌ | ❌ | admin_required |
| 匯入 Excel 人員 | ✅ | ❌ | ❌ | ❌ | admin_required |
| 管理條文設定 | ✅ | ❌ | ❌ | ❌ | admin_required |

## 三、登入與帳號安全機制

### 3.1 登入流程

| 步驟 | 說明 |
|------|------|
| 1 | 使用者輸入員工編號與密碼 |
| 2 | 查詢 employee_id → 找到則用；否則 fallback 查詢 username（舊帳號相容） |
| 3 | 帳號是否存在？否 → flash 錯誤（不揭示是帳號還是密碼錯誤，防止帳號列舉攻擊） |
| 4 | 帳號是否啟用（is_active）？否 → flash「帳號已停用，請聯繫管理員」 |
| 5 | 帳號是否鎖定（locked_until > now）？是 → flash 剩餘鎖定分鐘數 |
| 6 | 密碼是否正確？否 → failed_login_count +1；若 ≥ 5 → locked_until = now + 30min |
| 7 | 登入成功 → 重置失敗計數、記錄 last_login、寫入 ActivityLog（含 IP 位址） |
| 8 | must_change_password == True → 強制 redirect 到 /change-password（不可繞過） |
| 9 | 正常登入 → redirect 到 dashboard |

### 3.2 帳號鎖定規則

| 設定 | 值 |
|------|----|
| 最大失敗次數（MAX_FAILED_ATTEMPTS） | 5 次 |
| 鎖定持續時間（LOCKOUT_MINUTES） | 30 分鐘 |
| 鎖定控制欄位 | `locked_until`（DateTime 或 None） |
| Admin 手動解鎖 | 後台將 locked_until = None、failed_login_count = 0 |

### 3.3 首次登入強制改密碼

- 新帳號建立時：`must_change_password = True`，初始密碼 = 員工編號
- 登入成功後若 `must_change_password == True`，強制 redirect 到 `/change-password`
- 改密碼成功後：`must_change_password = False`
- 使用者無法繞過此步驟（即使直接輸入 URL 也會被 redirect 攔截）

### 3.4 密碼強度規則（v1.7.0 強化 ★）

| # | 規則 | 說明 |
|---|------|------|
| 1 | 最短 8 個字元 | 基本長度要求 |
| 2 | 至少 1 個大寫英文字母（A–Z） | 增加密碼複雜度 |
| 3 | 至少 1 個小寫英文字母（a–z） | 增加密碼複雜度 |
| 4 | 至少 1 個數字（0–9） | 增加密碼複雜度 |
| 5 | 不得與員工編號相同 | 防止初始密碼未更換的情形 |
| 6 | 不得為常見弱密碼黑名單 | 例：Password1、Admin1234、Hospital1 等 |

> 實作位置：`helpers.py` 新增 `validate_password_strength()` 共用函數，供 `change_password` 與 `profile` 路由引用。

## 四、安全強化改善計畫（v1.7.0 新增）

### 優先級總覽

| # | 優先 | 改善項目 | 難度 | 影響範圍 |
|---|------|----------|------|----------|
| 1 | 🔴 高 | SECRET_KEY 正式環境強制驗證 | 低（改 config.py） | 部署安全，防止預設金鑰被用於正式環境 |
| 2 | 🔴 高 | 密碼強度驗證升級 | 低（改 helpers.py + auth.py） | 帳號安全，符合醫療資訊系統最低標準 |
| 3 | 🟡 中 | 督導看不到資料的診斷提示機制 | 低（改 permissions.py + 前端） | 使用者體驗，解決督導無聲失敗問題 |
| 4 | 🟢 低 | 姓名比對 → user_id 外鍵改造 | 高（需 DB migration） | 資料正確性，解決同名同姓風險（長期根本方案） |

### 4.1 SECRET_KEY 正式環境強制驗證

**問題：** `config.py` 中存在預設明文金鑰。若管理員忘記設定環境變數，正式環境就使用了不安全的預設值，任何人只要看到原始碼就能偽造 Session。

**修正方案：**
- `ProductionConfig` 改用 `@property` 動態讀取 `SECRET_KEY`
- 若環境變數未設定，直接拋出 `RuntimeError` 並顯示操作指引，阻止系統啟動
- `run.py` 啟動時加入提醒訊息

**產生安全金鑰的指令：**
```bash
python -c "import secrets; print(secrets.token_hex(32))"
```

### 4.2 密碼強度驗證升級

**問題：** 目前版本只要求 8 個字元，對存有病患進度、評鑑文件的醫療系統強度不足。

**修正方案：**
- `helpers.py` 新增 `validate_password_strength(password)` 共用函數，回傳 `(bool, error_message)`
- 套用位置：`change_password` 路由（首次強制改密碼）和 `profile` 路由（自主改密碼）
- 前端輸入欄下方加入即時提示文字

### 4.3 督導看不到資料的診斷提示機制

**問題：** 當督導的 `supervisor_name` 欄位未設定時，系統靜默回傳 False，督導登入後看到空白儀表板，不知道原因。

**修正方案：**
- `permissions.py` 新增 `get_supervisor_diagnostic(user)` 函數
- 回傳：匹配表格數、比對使用姓名、未設定督導的表格數
- `overview.html` 頁面：若 supervisor 角色且匹配表格數為 0，顯示醒目警示框

**提示訊息設計：**

| 情境 | 顯示內容 |
|------|----------|
| 匹配 ≥ 1 張 | 正常顯示，不顯示警示框 |
| 匹配 = 0（有 real_name） | ⚠️ 「系統以姓名「XXX」進行比對，目前未找到您負責督導的表格，請確認督導欄位設定或聯絡品管中心」 |
| real_name 未設定 | ❌ 「您的帳號尚未設定真實姓名，請聯絡系統管理員更新帳號資料」 |

### 4.4 姓名比對 → user_id 外鍵改造（長期方案）

**問題：** 同名同姓情況下，使用者 A 可能看到使用者 B 負責的表格。

**階段一（v1.7.0 短期）：強化 warning log**
- `can_edit_table()` 在純姓名比對成功時，記錄 WARNING log 供管理員定期審查
- log 格式：`[權限警告] 使用者 EMP001（李OO / 護理部）透過純姓名比對存取表格 ID=xxx`

**階段二（v1.8.0 長期）：資料庫外鍵改造**
- `AuxiliaryTable` 新增 `form_filler_user_id` 和 `supervisor_user_id` 外鍵欄位（`nullable=True`，向下相容）
- `can_edit_table()` 優先使用 `form_filler_user_id` 比對；若外鍵為 NULL 則降級使用姓名比對
- Admin 後台提供「外鍵對應工具」：自動掃描現有姓名，建議對應 user.id，管理員確認後批次更新

**Migration SQL（v1.8.0）：**
```sql
ALTER TABLE auxiliary_tables
  ADD COLUMN form_filler_user_id INTEGER REFERENCES users(id),
  ADD COLUMN supervisor_user_id  INTEGER REFERENCES users(id);

CREATE INDEX idx_at_filler_user ON auxiliary_tables(form_filler_user_id);
CREATE INDEX idx_at_supervisor   ON auxiliary_tables(supervisor_user_id);
```

## 五、使用者管理（Admin 後台）

### 5.1 功能清單

> 頁面路徑：`/users`（需 `admin_required`）

| 功能 | 路由 | 說明 |
|------|------|------|
| 列出所有使用者 | `GET /users` | 頁面，含「配對」欄顯示匹配表格數 |
| 使用者 JSON 清單 | `GET /users/list-json` | Alpine.js 呼叫，含 matched_tables_count |
| 新增使用者 | `POST /users/create` | 表單建立，初始密碼 = 員工編號 |
| 更新使用者 | `PUT /users/<id>` | 修改角色、real_name、department 等 |
| 重設密碼 | `POST /users/<id>/reset-password` | 密碼重設為員工編號，must_change_password = True |
| 切換啟用狀態 | `POST /users/<id>/toggle-active` | 停用 / 啟用（不刪除帳號，保留歷史紀錄） |
| 從負責人名單建立 ★ | `GET /users/extract-from-db` | Plan C 抽取 API |
| 批次建立 ★ | `POST /users/batch-create-from-extract` | Plan C 批次建立 API |

> 內建的 `admin` 帳號（`username='admin'`）永久無法被停用、刪除或降權。

### 5.2 Plan C：從負責人名單建立帳號

**設計目標：** 系統已有大量填表人和督導長官的姓名資料，透過掃描這些欄位，自動提案需要建立帳號的人員清單，由 Admin 補填員工編號後批次建立帳號。

**抽取流程（GET /users/extract-from-db）：**
1. 掃描 `auxiliary_tables`（`form_filler_name`、`supervisor_name`）
2. 掃描 `articles` 和 `task_subs` 中的填表人欄位
3. 使用 `split_filler_name()` 拆分斜線分隔的多人名稱
4. 使用 `clean_name()` 清理名稱（去全形空格、排除非人名字串）
5. 依（real_name, department）去重
6. 偵測角色衝突（同一人同時出現在填表人和督導名單）
7. 比對現有 users 資料表，標記 already_exists

**前端 Modal 功能設計：**
- 摘要統計列：總計 / 未建立 / 衝突數
- 篩選按鈕：全部 / 未建立 / 督導長官 / 填表人 / ⚠️ 衝突
- 可勾選清單，含「角色」下拉可調整
- 衝突列：橘色左邊框（`border-l-4 border-amber-400`）標示
- 已存在帳號：灰色（`opacity-60`）顯示

### 6.2 匯入三步驟流程

| 步 | 動作 | 路由 / 說明 |

| 1 | 下載範本 | `GET /api/import-contacts/template` → 產生 24 欄 Excel（A–T），含底色分組、欄位說明、範例資料列 |

### 6.3 帳號自動建立邏輯（步驟 3）

| 情境 | 處理方式 |
|------|----------|
| 員工編號欄有值，帳號不存在 | 自動建立帳號：初始密碼 = 員工編號，must_change_password = True，角色依欄位（D填表人1 / H填表人2 / M負責人 → filler；R督導 → supervisor） |
| 員工編號欄有值，帳號已存在 | 更新 real_name / phone / email（若有變更），不更改角色 |
| 員工編號欄為空 | 不建立帳號，只寫入姓名 / 電話到表格欄位 |
| 角色衝突（D 欄和 R 欄同一人） | 優先採用較高權限（supervisor），預覽時顯示 ⚠️ 衝突警告 |
| 帳號已有更高角色 | 不強制降權，預覽顯示「此帳號目前為 supervisor，匯入不更改角色」 |

### 6.4 同一人同時為填表人1 與填表人2

若同一 `employee_id` 同時出現在 D 欄和 H 欄：
- 視為同一個 filler 帳號，建立一次即可
- 預覽時顯示提示「填表人1 與填表人2 為同一人，將建立一個 filler 帳號」
- 表格同時記錄 `form_filler_name`（填表人1）和 `co_filler_name`（填表人2）

### B.1 validate_password_strength()（helpers.py 新增）

```python
import re

def validate_password_strength(password):
    if len(password) < 8:
        return False, '密碼至少需要 8 個字元'
    if not re.search(r'[A-Z]', password):
        return False, '密碼必須包含至少一個大寫英文字母'
    if not re.search(r'[a-z]', password):
        return False, '密碼必須包含至少一個小寫英文字母'
    if not re.search(r'\d', password):
        return False, '密碼必須包含至少一個數字'
    weak = {'Password1', 'Admin1234', 'Hospital1', 'P@ssw0rd'}
    if password in weak:
        return False, '密碼強度不足，請勿使用常見密碼'
    return True, None
```

### B.2 get_supervisor_diagnostic()（permissions.py 新增）

```python
def get_supervisor_diagnostic(user):
    from app.models import AuxiliaryTable
    if not user.has_permission('assess') or user.has_permission('all'):
        return None
    if not user.real_name:
        return {'matched_count': 0,
                'reason': '帳號未設定真實姓名，請聯絡管理員'}
    all_tables = AuxiliaryTable.query.all()
    matched = [t for t in all_tables
               if t.supervisor_name and
               t.supervisor_name.strip() == user.real_name.strip()]
    return {
        'matched_count': len(matched),
        'user_real_name': user.real_name,
        'reason': f'以姓名「{user.real_name}」比對督導欄位，找到 {len(matched)} 張表格'
    }
```
